# 分析场景模型优化方案

## 背景

当前 API 接口 `/api/v1/parse` 调用时需要手动传入 `dbc_file` 和 `signal_filter_list` 参数。产品形态上需要引入"分析场景模型"概念，将车型 + 模型与 dbc 文件、信号过滤列表的映射关系预定义好，接口调用时只需传入车型和模型名称即可自动完成数据解析。

## 核心思路

引入 **车型（Vehicle Model）** 和 **分析场景模型（Model）** 两级概念：

- **车型**：车辆型号（如 C01、C11、T03），一个车型对应一组 DBC 文件
- **模型**：该车型下的分析场景（如底盘分析、车身电子分析），一个模型对应一组 DBC 文件 + 信号过滤列表

接口调用只需传入 `vehicle_model` + `model_name`，系统自动查找对应的配置。

## 新增文件结构

```
src/can_parser_server/
├── config/
│   └── models_config.json        # 车型→模型→DBC+信号列表的映射配置
├── dbc_files/                    # 本地维护的DBC文件目录（按车型分子目录）
│   ├── C01/
│   │   ├── vehicle_chassis.dbc
│   │   ├── vehicle_body.dbc
│   │   └── ...
│   ├── C11/
│   │   ├── vehicle_chassis.dbc
│   │   └── ...
│   └── ...
└── api_server.py                 # 接口（修改）
```

## 配置文件设计 (`models_config.json`)

```json
{
  "vehicles": {
    "C01": {
      "description": "C01 车型",
      "models": {
        "chassis_analysis": {
          "dbc_files": ["vehicle_chassis.dbc", "vehicle_powertrain.dbc"],
          "signal_filter_list": ["VehicleSpeed", "SteeringAngle", "BrakePressure", "MotorTorque"],
          "description": "底盘分析场景"
        },
        "body_electronics": {
          "dbc_files": ["vehicle_body.dbc"],
          "signal_filter_list": ["DoorStatus", "WindowPosition", "LightStatus"],
          "description": "车身电子分析场景"
        }
      }
    },
    "C11": {
      "description": "C11 车型",
      "models": {
        "chassis_analysis": {
          "dbc_files": ["vehicle_chassis.dbc"],
          "signal_filter_list": ["VehicleSpeed", "SteeringAngle", "SuspensionHeight"],
          "description": "底盘分析场景"
        }
      }
    }
  }
}
```

- `dbc_files` 的值是相对于 `dbc_files/{vehicle_model}/` 目录的文件名数组
- 每个模型可关联多个 DBC 文件，解析时依次加载所有 DBC 文件的信号定义
- `signal_filter_list` 直接定义该模型关注的信号列表
- 不同车型的同名模型可以使用不同的 DBC 文件和信号列表

## 接口改造

**现有接口参数：**
```
parser_type, data_file, dbc_file(可选), signal_filter_list(可选)
```

**改造后新增参数：**
```
vehicle_model: Optional[str]  # 车型（如 C01、C11）
model_name: Optional[str]     # 分析场景模型名称
```

**逻辑优先级：**
- 传入 `vehicle_model` + `model_name` → 从配置文件查找 `vehicles[vehicle_model].models[model_name]`，自动获取 `dbc_files` 路径列表和 `signal_filter_list`，忽略上传的 `dbc_file` 参数
- 未传入 `model_name` → 走现有逻辑，手动上传 `dbc_file` + 可选的 `signal_filter_list`（保持向后兼容）
- csv 类型不需要 dbc，`model_name` 仅指定 `signal_filter_list`

## 配置加载方式

在 `api_server.py` 启动时加载 `models_config.json` 到内存中的一个 dict，避免每次请求都读文件：

```python
import json
from pathlib import Path

CONFIG_DIR = Path(__file__).parent / "config"
DBC_DIR = Path(__file__).parent / "dbc_files"

def load_models_config():
    with open(CONFIG_DIR / "models_config.json") as f:
        return json.load(f)["vehicles"]

MODELS = load_models_config()
```

## 新增辅助管理接口

| 接口 | 说明 |
|---|---|
| `GET /api/v1/models` | 列出所有车型及其模型概览（名称和描述） |
| `GET /api/v1/models/{vehicle_model}` | 查看某车型下所有模型的详情（dbc 文件名列表和 signal_filter_list） |

方便调用方了解有哪些车型和模型可用，而不需要去翻配置文件。

## 改造点汇总

| 改动项 | 说明 |
|---|---|
| 新增 `config/models_config.json` | 车型→模型→DBC+信号列表的二级映射配置，线下维护 |
| 新增 `dbc_files/` 目录 | 按车型分子目录存放本地 dbc 文件 |
| 修改 `/api/v1/parse` 接口 | 新增 `vehicle_model` + `model_name` 参数，自动解析 dbc + signal_filter_list |
| 解析逻辑调整 | `model_name` 存在时，从 `dbc_files/{vehicle_model}/` 读取对应 dbc 文件路径，从配置获取 `signal_filter_list` |
| 新增 `/api/v1/models` 接口 | 所有车型及模型概览 |
| 新增 `/api/v1/models/{vehicle_model}` 接口 | 某车型下模型详情 |

## 关键设计决策

- **车型 + 模型两级结构**：同一车型下不同分析场景共享 DBC 文件目录，不同车型的 DBC 文件物理隔离
- **向后兼容**：vehicle_model 和 model_name 为可选参数，不传就保持原有方式，不影响现有调用方
- **配置与代码分离**：模型映射放在 JSON 配置文件中，方便运营人员线下维护，不需要改代码
- **DBC 文件按车型组织**：`dbc_files/{vehicle_model}/` 目录结构，车型间物理隔离，清晰易维护
- **CSV 类型的处理**：CSV 不需要 dbc 文件，但 model_name 仍可用来获取 signal_filter_list（如果有的话）
