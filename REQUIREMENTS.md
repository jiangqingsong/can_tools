# 需求文档（REQUIREMENTS.md）

## 1. 项目背景

车辆 CAN 总线产生的日志数据需要被解码、结构化、入库，才能供数据分析人员查询使用。当前缺少一个统一的服务来承载这一流程——数据工程师需要手动处理不同格式（ASC、BLF、CSV）的 CAN 日志文件，效率低且容易出错。

## 2. 核心需求

### REQ-01：支持多种 CAN 日志格式解析

| 项目 | 内容 |
|------|------|
| 优先级 | P0（已实现） |
| 描述 | 服务需支持 ASC、BLF、CSV/Excel 三种常见 CAN 日志格式的解析 |
| 验收标准 | 上传任意支持的格式文件，服务能正确解析并返回写入条数 |
| 关联设计 | DESIGN.md §5.2, 5.3, 5.4 |

### REQ-02：DBC 信号解码

| 项目 | 内容 |
|------|------|
| 优先级 | P0（已实现） |
| 描述 | ASC/BLF 格式需配合 DBC（CAN Database）描述文件，将原始 CAN 报文（arbitration_id + payload）解码为可读的信号名和物理值 |
| 验收标准 | 上传 ASC/BLF + DBC 文件，解析结果中信号名和物理值正确，与 CANoe/CANalyzer 等工具解码结果一致 |
| 关联设计 | DESIGN.md §5.2, 5.3 |

### REQ-03：信号过滤

| 项目 | 内容 |
|------|------|
| 优先级 | P1（已实现） |
| 描述 | 支持按信号名称过滤，仅解析和存储用户关心的信号子集，减少数据量和处理时间 |
| 验收标准 | 传入 `signal_filter_list=["VehicleSpeed"]`，仅返回 VehicleSpeed 信号数据，其他信号被跳过 |
| 关联设计 | DESIGN.md §3.1 |

### REQ-04：数据写入 StarRocks

| 项目 | 内容 |
|------|------|
| 优先级 | P0（已实现） |
| 描述 | 解析后的结构化信号数据写入 StarRocks 分析数据库，供下游查询和分析 |
| 验收标准 | 解析完成后，数据可通过 SQL 在 StarRocks 中查询，条数与解析返回的 `total_written` 一致 |
| 关联设计 | DESIGN.md §4.1, 5.5, 5.6 |

### REQ-05：异步任务执行

| 项目 | 内容 |
|------|------|
| 优先级 | P0（已实现） |
| 描述 | 文件解析可能耗时较长（大文件可达数分钟），接口需采用异步模式：提交任务立即返回 task_id，调用方通过 task_id 轮询状态和结果 |
| 验收标准 | POST /parse 立即返回（不等待解析完成），GET /tasks/{id} 可查到 pending→running→success/failed 的完整状态流转 |
| 关联设计 | DESIGN.md §3.1, 3.2 |

### REQ-06：并发控制

| 项目 | 内容 |
|------|------|
| 优先级 | P1（已实现） |
| 描述 | 限制同时解析的任务数（当前为 4 个），超出限制的任务排队等待，防止资源耗尽 |
| 验收标准 | 同时提交 6 个任务，前 4 个 running，后 2 个 pending；有任务完成后 pending 任务自动进入 running |
| 关联设计 | DESIGN.md §5.1 |

### REQ-07：健康检查

| 项目 | 内容 |
|------|------|
| 优先级 | P2（已实现） |
| 描述 | 提供健康检查端点，供运维监控服务存活状态 |
| 验收标准 | GET /health 返回 `{"status": "ok"}` 和当前支持的文件类型列表 |
| 关联设计 | DESIGN.md §3.3 |

## 3. 待规划需求

### REQ-08：分析场景模型

| 项目 | 内容 |
|------|------|
| 优先级 | P1（待开发） |
| 描述 | 引入"分析场景模型"概念，将 DBC 文件列表 + 信号过滤列表的映射关系预定义为车型 + 模型配置。调用方仅需传入 `vehicle_model`（车型）和 `model_name`（模型名称），系统自动查找对应的一组 DBC 文件和信号列表，无需每次手动上传 DBC 文件 |
| 验收标准 | 传入 `vehicle_model="C01"` + `model_name="chassis_analysis"` 即可完成解析（自动加载该模型关联的多个 DBC 文件），不传 dbc_file 参数；不传 model_name 时原有流程不受影响 |
| 关联文档 | `分析场景模型优化方案.md`，DESIGN.md §10，DEVELOPMENT_PLAN.md Phase 6 |

## 4. 非功能性需求

| 编号 | 需求 | 说明 |
|------|------|------|
| NF-01 | 凭据安全 | 数据库用户名/密码仅通过环境变量注入，不写入代码或配置文件 |
| NF-02 | 文件清理 | 上传的临时文件在任务完成后自动清理 |
| NF-03 | 容错写入 | 部分数据行写入失败不阻塞整体流程（max_filter_ratio=1.0） |
| NF-04 | 连接重试 | StarRocks 连接失败时自动重试（最多 5 次，退避延迟） |
| NF-05 | 编码兼容 | CSV 文件需兼容 utf-8、gbk、gb2312、gb18030、latin-1 等常见编码 |
